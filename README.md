# CS-320-Software-Testing
SNHU Software Testing course projects


How can I ensure that my code, program, or software is functional and secure?

Through manual code review, best-coding practices (such as appropriate naming conventions, in-line commenting,
appropriate data types and structures, encapsulation, etc.), and automated or manual component (or integration/system) testing which 
monitors correct and incorrect input, as well as output returned. Using a well documented and structure business requirements document, 
requirements of components or systems can be built into a program then checked for accuracy and acceptance. Using a completion percentage can help
reduce any defects and find bugs that need to be fixed before each component is complete, helping to reduce any risks.

How do I interpret user needs and incorporate them into a program?
User needs are generally determined in system analysis and incorporated into the system design. Interpretting user needs
is presented in the form of requirements (documented in the analysis) and can be built into the system, then checked and tested for
accuracy/precision. Using test completion percentage can help ensure that all user requirements have been met, exist, and function
appropriately within the program.

How do I approach designing software?
Software designing starts with gathering and interpretting the requirements, documented and diagrammed (if necessary), and built into the components or system.
After they are built, they are tested and reviewed (fixed if needed), then implemented (deployed) and monitered for further bugs or defects. This is the basic 
concept of the software development lifecycle (SDLC). A program design is built to sufficiently meet the requirements of the user or the system, and often requires many iterations to ensure successful completion and deployment of the components or system.
